# P1: CI Integration Changes

## Date: 2026-06-15

## Summary

Three changes to make ActionD CI results visible in the terminal after `lgh up`:

1. **Auto-start ActionD** when LGH starts
2. **Event ID tracking** to replace sleep+substring job matching
3. **Terminal CI result display** after `lgh up`

## Changes

### P1.1: Auto-start ActionD

**Files**: `lgh/cmd/lgh/serve.go`, `lgh/internal/server/server.go`

LGH server now has an `OnReady` callback that fires after the IPC socket is created
but before `ListenAndServe`. The `tryStartActionD()` function:

1. Collects candidate paths: PATH â†’ `~/.local/bin/actiond` â†’ repo binary
2. Validates each with `actiond version` (exit 0) â€” skips SIGKILL'd binaries
3. Checks if ActionD PID file exists and process is alive (`kill -0`)
4. If not running, executes `actiond start -d`
5. Failures logged to stderr, never block LGH startup

**Key design**: The callback fires synchronously in `srv.Start()` after `startIPC()`,
ensuring the Unix socket exists before ActionD tries to connect.

### P1.2: Event ID Tracking

**Files**: 
- `ActionD/internal/store/memory.go` â€” added `ListJobsByEventID` to Store interface
- `ActionD/internal/store/sqlite.go` â€” SQLite implementation with `WHERE event_id = ?`
- `ActionD/internal/server/server.go` â€” `GET /api/actions/by-event/{event_id}` endpoint
- `ActionD/internal/mcp/client.go` â€” `GetActionsByEventID()` HTTP client method
- `lgh/internal/mcp/tools.go` â€” `findEventIDForCommit()` + `pollActionDByEventID()`

**Old flow** (fragile):
```
lgh up â†’ sleep 3s â†’ GET /api/actions?limit=15 â†’ substring match by repo name
```

**New flow** (reliable):
```
lgh up â†’ read event log â†’ find event_id by commit hash â†’ poll /api/actions/by-event/{event_id}
```

The event_id (UUID) is generated by LGH on each push and stored as `ActionJob.EventID`
in ActionD. This is the reliable link between push and CI jobs.
